# Elevation Calibration Notes

This document records the deliberate "magic" in elevation analysis heuristics.
These values are not physical constants. They are calibration choices, backed by
the current product requirements, unit tests, and representative internal
calibration scenarios.

This file explains why calibrated numbers were chosen. The end-to-end
processing order is specified in `docs/elevation-model.md`. Fixture scenarios
and tolerance policy are documented in `docs/elevation-fixtures.md`.

When changing any value below, update this document, `docs/elevation-model.md`
if processing order changes, `docs/product-spec.md` if user-visible behavior
changes, and the relevant regression tests.

## Documentation Placement

Keep documentation at the level where it helps most:

- Code comments should stay short and local. Use them beside calibrated
  constants or non-obvious branches to explain why the code is shaped that way
  and to point back to this file.
- This file is the elevation calibration register. It owns numeric thresholds,
  trust weights, rationale, external references when they are verified, and
  regression guardrails.
- `docs/product-spec.md` should describe user-visible behavior and compatibility
  expectations, not every numeric threshold.
- Tests are the executable form of calibration. If a threshold changes, add or
  update a focused regression that explains the intended behavior.

## Calibration Sources

The current values come from several kinds of evidence:

- robust-statistics conventions: MAD scaled by `1.4826`, Hampel-style
  outlier rejection, and sigma multipliers;
- source behavior assumptions: barometric data is usually better for local
  relative changes, terrain data is better as an absolute anchor, and raw
  GPX/GNSS elevation is often vertically noisy;
- synthetic invariants in tests: sampling density should not materially change
  gain/loss, isolated spikes should not define filtered extrema, and water-like
  tracks should suppress short vertical oscillations.

## Activity Inference

Implementation: `src/core/activity-inference.js`.

The activity classifier is intentionally coarse. It chooses parameters for the
elevation model; it is not a general-purpose sport classifier.

Activity inference is a coarse routing signal. It must expose ambiguity through
diagnostics rather than pretending speed and elevation heuristics are a complete
sport classifier.

When a parser provides explicit structured activity provenance, the elevation
model uses that activity before speed/elevation inference. This bypass has
confidence `0.95`, reason code `explicit_activity`, and still records feature
diagnostics for transparency.

Known risk: activity inference is intentionally coarse. Running, e-scooter,
urban motor traffic, slow mountain biking, untimed recorded tracks, and noisy
long descents can be ambiguous. The model should expose ambiguity diagnostics
before changing selected activity rules.

## Source Assessment

Implementation: `src/core/elevation-source-assessor.js`.

The source assessor separates absolute trust (`absTrust`) from relative trust
(`relTrust`). These scores are calibrated weights, not probabilities.

| Value                                                                                                                | Where                                                                                       | Origin                                                                        | Why                                                                                                                                         |
| -------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Dense pair distance `<= 80 m`                                                                                        | `DENSE_PAIR_MAX_DISTANCE_METERS`                                                            | Recording-density guard.                                                      | Large vertical deltas in dense samples are much more suspicious than the same deltas in sparse mountain samples.                            |
| Implausible grade `>= 0.35`                                                                                          | `ELEVATION_IMPLAUSIBLE_GRADE` in `src/core/elevation-calibration-constants.js`              | Conservative vertical-jump guard.                                             | A 35% point-to-point grade is high enough to treat a large vertical step as likely sensor noise unless other context says otherwise.        |
| Chatter grade `>= 0.18`                                                                                              | `CHATTER_GRADE`                                                                             | Low-relief noisy GPS calibration.                                             | Repeated dense deltas around this grade can inflate gain/loss even when each individual step is smaller than the implausible cutoff.        |
| Implausible vertical speed `>= 1.2 m/s`                                                                              | `ELEVATION_IMPLAUSIBLE_VERTICAL_SPEED_MPS` in `src/core/elevation-calibration-constants.js` | GNSS/barometric vertical-speed guard.                                         | Large elevation deltas over short time intervals are more likely height-channel noise than traveled-surface relief.                         |
| Noisy if dense p75 absolute delta `>= 2 m` or p95 delta `>= 8 m` with high grade/speed context                       | `denseChatter`                                                                              | Vertical-channel noise calibration from GPX-shaped fixtures and source tests. | Frequent point-to-point jumps inflate naive gain/loss, but sparse plausible mountain grades are not downgraded only because deltas are big. |
| Non-barometric noisy if raw-change-to-range ratio `>= 12` and dense/steep context                                    | `highChangeRatioNoise`                                                                      | Low-relief noisy GPS calibration.                                             | If total absolute vertical motion is far larger than the actual height envelope, the signal is likely oscillating.                          |
| Barometric low-relief chatter if range `<= 1 m` and ratio `>= 8`                                                     | `lowReliefChatter`                                                                          | Flat/near-flat barometric guard.                                              | Prevents tiny pressure or quantization ripple from becoming high relative trust in very low relief.                                         |
| Barometric drift possible if run length `>= 4`, range `>= 8 m`, p95 delta `< 2 m`, directional consistency `>= 0.85` | `hasSlowBarometricTrend`                                                                    | Barometer behavior assumption.                                                | A smooth one-direction trend can be good relative shape but suspicious absolute level because pressure drift can mimic slow ascent/descent. |
| Barometric spike run if p75 `>= 2 m` or p95 `>= 8 m`                                                                 | `hasBarometricSpikeRun`                                                                     | Same vertical spike thresholds as source noise.                               | Keeps barometric relative trust high in normal cases but reduces it when the barometer contains repeated jumps.                             |

Trust-score calibration:

| Source case                         | `absTrust` |                  `relTrust` | Why                                                                                                                           |
| ----------------------------------- | ---------: | --------------------------: | ----------------------------------------------------------------------------------------------------------------------------- |
| Clean barometric                    |     `0.55` |                      `0.90` | Barometer is usually strong for local relative shape, but absolute level can still need calibration.                          |
| Barometric with noise/drift/chatter |     `0.35` |            `0.55` or `0.90` | Absolute level is downgraded more aggressively than relative signal; spikes downgrade relative trust to `0.55`.               |
| Terrain                             |     `0.78` | `0.25` noisy / `0.45` clean | Terrain is a good absolute anchor but can be poor for local traveled-surface detail, bridges, tunnels, forest, and buildings. |
| Clean GPX/GNSS                      |     `0.55` |                      `0.60` | Raw GPX can be usable when local noise is low.                                                                                |
| Noisy GPX/GNSS                      |     `0.35` |                      `0.25` | Noisy GPS height should not drive gain/loss sensitivity.                                                                      |
| Clean unknown                       |     `0.45` |                      `0.50` | Unknown source is usable but less trusted than declared GPX/barometric/terrain.                                               |
| Noisy unknown                       |     `0.25` |                      `0.20` | Unknown noisy height is the least trusted case.                                                                               |

## Fusion And Cleanup

Implementation: `src/core/elevation-fusion.js`.

The Hampel spike floor is relief-aware. The historical `12 m` floor is still the
maximum floor, but low-relief runs can use a lower floor so an isolated 8-10 m
spike is filtered on flat water, city, or low-relief cycling routes.

## Extrema And Confidence

Implementation: `src/core/elevation-profile.js`.

| Value                                                                  | Where                                 | Origin                                                            | Why                                                                                                                                                            |
| ---------------------------------------------------------------------- | ------------------------------------- | ----------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Filtered extrema are clamped to a robust raw envelope                  | `limitExtremaToRobustRawEnvelope`     | Product requirement for user-visible min/max.                     | The chart, gain/loss, and min/max should not contradict each other because of one raw spike.                                                                   |
| Per-run supported confirmed extrema can relax the raw-envelope clamp   | `getSupportedConfirmedProfileExtrema` | Chart/metric consistency regression.                              | A short hill or trough confirmed for gain/loss must remain visible when same-run raw support exists; severe isolated raw spikes remain clamped.                |
| Confirmed-extrema raw-support guard `max(60 m, neighborDelta * 3)`     | `isSevereIsolatedRawOutlier`          | Reuses the shared time-gap discontinuity scale for severe spikes. | Lets confirmed short hills/troughs relax the robust envelope while preventing a raw `1000 m`-style isolated spike from becoming a visible extrema.             |
| Isolated raw outlier threshold `max(12 m, neighborDelta * 3)`          | `replaceIsolatedRawOutliers`          | Same spike floor and 3x local-neighbor rule as fusion cleanup.    | Keeps real local slopes while removing a lone point far away from both neighbors.                                                                              |
| Low activity confidence flag below `0.5`                               | `buildConfidenceFlags`                | Diagnostic threshold.                                             | Marks cases where activity defaults may be weak without blocking calculation.                                                                                  |
| Source-switch fragmentation penalty `0.10`                             | `buildConfidencePenalties`            | Mixed-source fragmentation guard.                                 | Frequent source alternation creates many run boundaries, so gain/loss should be lower confidence.                                                              |
| Many short elevation runs penalty `0.10`                               | `buildConfidencePenalties`            | Fragmented-run guard.                                             | Short runs limit Hampel/smoothing context and can preserve endpoint artifacts.                                                                                 |
| Outlier replacement penalty `0.05` / high penalty `0.15`               | `buildConfidencePenalties`            | Cleanup-quality diagnostic.                                       | A profile that needed many replacements is usable but should not look as reliable as a clean profile.                                                          |
| Endpoint spike replacement penalty `0.05`, many endpoint spikes `0.12` | `buildConfidencePenalties`            | Endpoint-specific cleanup diagnostic.                             | Endpoint replacements can affect visible extrema, net change, and first/last legs, so they are surfaced separately from ordinary interior Hampel replacements. |
| Activity ambiguity penalty `0.08`                                      | `buildConfidencePenalties`            | Activity-candidate diagnostic.                                    | Near-tied activity candidates mean thresholds may be sensitive to the coarse classifier.                                                                       |
| Overall confidence `(gainLoss * 2 + extrema) / 3`                      | `buildConfidence`                     | Product weighting.                                                | Gain/loss quality matters more to the elevation model than filtered extrema quality.                                                                           |

## Current Guardrails

The most important tests protecting these calibration values are:

- `tests/unit/activity-inference.test.js`
- `tests/unit/elevation-source-assessor.test.js`
- `tests/unit/elevation-fusion.test.js`
- `tests/unit/elevation-gain-integrator.test.js`
- `tests/unit/elevation-profile.test.js`
- `tests/unit/track-analyzer.test.js`
- metric fixture checks run by `npm run test:metrics`

When recalibrating, prefer adding a real or synthetic regression that explains
the intended behavior before changing the number. The value itself can be magic;
the reason for the value should not be.
